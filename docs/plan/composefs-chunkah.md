# Deferred: ComposFS + Chunkah

## Why composefs matters

Composefs is a read-only filesystem that mounts the ostree commit as a content-addressed
overlay. Each file is a hard-link from a central object store (`/sysroot/.ostree/repo/objects/`).
With `fs-verity` enabled, every file's digest is verified on first access â€” this is the
tamper-resistance layer that makes secure boot meaningful end-to-end. Without composefs,
an attacker with physical access can modify the root filesystem after boot without
breaking any signature check.

Composefs support is also the prerequisite for `bootc install to-disk --composefs-backend`
to succeed; without it the install fails with "Invalid splitstream content type".

## How dakota implements it

Dakota uses **chunkah** (`quay.io/coreos/chunkah`) to convert the squashed OCI image into
a composefs-ready image. The pipeline:

1. `just export` â€” `sudo podman build --squash-all` produces a single-layer OCI image.
   (Multi-layer output breaks composefs xattr injection; see dakota#841.)

2. `just chunkify` â€” three sub-steps:
   a. Mount the squashed image as a writable overlay.
   b. Run `fakecap-restore` (a small C program compiled from `files/fakecap/fakecap-restore.c`)
      to physically apply `user.component` xattrs from `files/fakecap-manifest.tsv`.
      chunkah uses rustix raw syscalls for xattr reads (bypasses `LD_PRELOAD`), so real
      xattrs must exist on the overlay before chunkah runs. (coreos/chunkah#113)
   c. Run chunkah against the read-only overlay. Chunkah splits the image into up to 120
      layers keyed by `user.component` â€” one layer per BST element group â€” then loads the
      result back into podman.

3. `just generate-bootable-image` â€” now uses `--composefs-backend` because the image has
   valid ostree splitstreams.

## The fakecap-manifest.tsv

`files/fakecap-manifest.tsv` in dakota is ~989K lines: a committed TSV mapping every
file path in the installed image to its source BST element and a cache-busting interval:

```
/usr/bin/bash   components/bash.bst   monthly
/usr/lib/systemd/systemd   components/systemd.bst   monthly
```

Generated by dakota's "rechunk" process (git commit message: "regenerate chunkah filemap
and fakecap manifest") and committed to the repo. Updated whenever elements change.

The `interval` field (`monthly`/`weekly`/`daily`) controls chunkah's layer grouping:
files from the same element *and* same interval end up in the same layer, providing
natural delta compression for OTA updates.

## What krytis needs

### 1. Generate fakecap-manifest.tsv

After a successful `mise run lint` (image in `localhost/krytis:latest`), scan the image
and attribute each file to its source BST element. Probably done by:

- Using `bst artifact show --format` to get per-element file lists from the BST cache
- Or mounting the squashed image and cross-referencing the `usr/manifest.json`
  (our `oci/krytis/manifest.bst` artifact) to map package â†’ element â†’ files

This produces `files/fakecap-manifest.tsv`, which should be committed and updated via a
`mise run generate-fakecap-manifest` task whenever elements change significantly.

### 2. Port fakecap-restore.c

Copy `files/fakecap/fakecap-restore.c` from dakota verbatim (MIT licensed). It reads the
TSV and calls `lsetxattr(path, "user.component", component_value, ...)` on each file.

### 3. Add `mise run chunkify` task

Model directly on dakota's `chunkify` just target:

```bash
# Compile fakecap-restore if needed
gcc -O2 -o files/fakecap/fakecap-restore files/fakecap/fakecap-restore.c

# Mount squashed image as writable overlay
LOWER=$(sudo podman image mount localhost/krytis:latest)
# ... setup UPPER/WORK/MERGED overlay dirs ...
sudo mount -t overlay overlay -o "lowerdir=${LOWER},upperdir=${UPPER},workdir=${WORK}" "$MERGED"

# Apply user.component xattrs
sudo ./files/fakecap/fakecap-restore files/fakecap-manifest.tsv "$MERGED"

# Run chunkah
CHUNKAH_REF="quay.io/coreos/chunkah:v0.6.0@sha256:ff8b8b466a942ec6000445d4001fc661e2fc5a952ad9ee29b4de9ab09d1d1708"
sudo podman run --rm --pull never \
    --security-opt label=type:unconfined_t \
    -v "${MERGED}:/chunkah:ro" \
    -e "CHUNKAH_ROOTFS=/chunkah" \
    "$CHUNKAH_REF" build --max-layers 120 --prune /sysroot/ \
    --label ostree.commit- --label ostree.final-diffid- \
    | sudo podman load

# Re-tag to localhost/krytis:latest
```

### 4. Re-enable `--composefs-backend` in generate-disk

Once chunkify produces a valid chunked image, restore the flag:

```bash
bootc install to-disk \
    --composefs-backend \
    --via-loopback "/data/${DISK}" \
    ...
```

## Relationship to secure boot / UKI

These features compose naturally:

- Composefs + fs-verity: verifies the *root filesystem* on access
- UKI + secure boot: verifies the *kernel + initramfs + cmdline* at boot

A fully attested boot chain needs both. Implement composefs first (it's self-contained),
then add UKI signing on top. See `docs/plan/secure-boot-uki.md`.
